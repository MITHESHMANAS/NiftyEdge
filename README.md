# N100 Financial Intelligence Platform

**Status: Sprint 1 (Data Foundation) + Sprint 2 (Ratio Engine) + Sprint 3 (Screener + Peer Engine) — COMPLETE**

This is the ETL + analytics + screener foundation for the Nifty 100
Financial Intelligence Platform: a 13-table SQLite database
(`nifty100.db`) built from 7 core + 5 supplementary Excel datasets
covering 92 Nifty 100 companies, a 16-rule data-quality validator, a full
Ratio Engine computing 50+ KPIs for every company-year, and a screener +
peer-comparison engine (6 preset screeners, an 11-peer-group percentile
ranking system, and 92 radar charts). Sprints 4–6 (Health Score/Trend
Analytics, Dashboard/Reporting, Alerts/API/QA) are out of scope for this
delivery — see `Makefile` targets for their stub placeholders.

## Quick start (< 5 minutes)

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Configure environment (optional — sensible defaults are baked in)
cp config/.env.template .env

# 3. Build the database (Sprint 1)
make load
# or: python3 src/etl/loader.py

# 4. Run the Ratio Engine — populates financial_ratios (Sprint 2)
make ratios
# or: python3 src/analytics/populate_financial_ratios.py

# 5. Run the Screener + Peer Engine — screener_output.xlsx,
#    peer_comparison.xlsx, peer_percentiles table, radar charts (Sprint 3)
make screener
# or: python3 src/screener/run_sprint3.py

# 6. Run the test suite (lint + format check + tests + coverage gate)
make lint
make format
make test
# or individually: pytest tests/ -v --cov=src --cov-report=term-missing --cov-fail-under=80

# 7. Generate HTML/XML coverage reports
make coverage
# opens at htmlcov/index.html

# 8. Explore the data
python3 -c "
import sqlite3

print(open('notebooks/exploratory_queries.sql').read())
"
```

## Design decisions & notes for the next sprint owner

- **12 tables, not 10.** The spec text says "10 tables" in a few places
  but the dataset catalogue (Sections 5–6) lists 12 source files
  (7 core + 5 supplementary). This build creates one table per source
  file — `companies, profitandloss, balancesheet, cashflow, analysis,
  documents, prosandcons, sectors, stock_prices, market_cap,
  financial_ratios, peer_groups` — since that's what the FK diagram
  (Section 7.1) and every downstream module actually needs.

- **`financial_ratios` table is Sprint-1 source data, not yet
  Sprint-2 computed data.** This load writes `financial_ratios.xlsx`
  as-is. Module 2 (Sprint 2) will recompute this table from raw
  P&L/BS/CF per the spec's KPI formulas and overwrite it — the schema
  and column names already match what Sprint 2 needs.

- **Real-world messiness handled by `normalize_year()`:** the raw P&L/
  BS/CF `year` column contains `'Mar 2023'`, `'Dec-13'`, bare `'2013'`
  (assumed March FYE), `'Mar 2023 15'` / `'Mar 2016 9m'` (trailing junk,
  base period extracted), and genuinely non-fiscal-year values (`'TTM'`,
  `'2024.5'`) which are correctly rejected per DQ-07 rather than guessed
  at. All of this was found by inspecting the actual source files, not
  assumed from the spec.

- **8 tickers are legitimately orphaned** (present in P&L/BS/CF but
  absent from `companies.xlsx`): `ULTRACEMCO, UNIONBANK, UNITDSPR, VBL,
  VEDL, WIPRO, ZYDUSLIFE, ZOMATO`. Confirmed by direct inspection —
  not a normalisation bug. These are rejected per DQ-03 (FK integrity)
  and logged in `validation_failures.csv`.

- **DQ-13 (URL validity)** is implemented but off by default — the
  1,585 `documents.xlsx` URLs point to `bseindia.com`, which isn't on
  this environment's network allowlist, and a full HEAD-request sweep
  is slow. Enable via `VALIDATE_URLS=true` in `.env` in an environment
  with unrestricted egress; `loader.build_database(validate_urls=True)`
  wires it up.

## Sprint 2 — Ratio Engine: design decisions & notes for the next sprint owner

- **Row count base: union of all annual periods, not just March FYE.**
  `financial_ratios` needs ≥1,100 rows (exit criterion). Restricting to
  standard March-ending fiscal years only yields 1,028 rows (below
  target) because several companies have historical Dec/Sep/Jun-ending
  reports before switching to a March FYE. Using the full union of every
  reported annual period across P&L/BS/CF gives 1,155 rows. CAGR windows
  use *positional* (Nth-prior-report) offsets rather than calendar-exact
  year gaps as a result — a reasonable approximation given the source
  data's mixed FY-end conventions, documented here rather than hidden.

- **NaN vs None — a real bug, fixed.** Every ratio function is written
  to treat `None` as "missing input" and return `None` for undefined
  ratios. But values pulled out of pandas (via `pd.read_sql`, `row.get()`,
  `Series.iloc[]`) surface missing data as `float('nan')`, not `None` —
  and `nan is None` is `False`, so an unpatched function would silently
  do arithmetic on NaN instead of correctly returning "undefined". This
  was caught during development (the `composite_quality_score` column
  came out entirely NaN for a company's early years) and fixed by adding
  `src/analytics/utils.clean()`, which every formula function now runs
  its inputs through first. Covered by `TestNaNSafety` test classes in
  `tests/kpi/`.

- **`financial_ratios` schema now has 42 columns**, not the ~14 in the
  spec's bullet list — the spec's Day 12 list of "14+ KPI columns" is a
  minimum; the sprint goal explicitly targets 50+ KPIs, so the schema
  also includes every flag/label column the daily tasks describe (ROCE,
  ROA, net debt, high-leverage flag, ICR label/risk flag, all 9 CAGR
  columns + 3 edge-case flags, CFO quality score/label, CapEx intensity
  %/label, FCF conversion rate, capital allocation pattern, composite
  score) rather than only the pass-through/computed columns named in the
  "Populate financial_ratios Table" bullet.

- **`book_value_per_share` is derived, not pass-through.** Raw data has
  no shares-outstanding time series. Using the standard Indian
  equity-structure identity `shares_outstanding = equity_capital /
  face_value` (face_value is static per company), book value per share
  is computed per year as `(equity_capital + reserves) * face_value /
  equity_capital`.

- **A real source-data bug was found and correctly caught, not
  papered over.** `ratio_edge_cases.log` flags ROCE/ROE values in the
  thousands of percent for BEL, HAL, LT, INDIGO, and PNB. Traced to raw
  `balancesheet.xlsx`: these companies' balance-sheet figures are ~2-3
  orders of magnitude smaller than their P&L figures (e.g. BEL: sales
  ₹20,268 Cr but reserves ₹73 Cr) — a genuine scale/unit inconsistency
  in the source file, confirmed by inspecting the raw Excel directly, not
  a bug in the ratio formulas. Categorised `FORMULA_DISCREPANCY` in the
  log per the Day 13 spec.

- **TCS's source `roe_percentage` (0.52%) is confirmed anomalous** — the
  spec calls this out explicitly, and the ratio engine's computed value
  (~50.9%, matching TCS's known real-world ROE) is used for all
  analytics; the source field is display-only. See
  `notebooks/day12_manual_spot_check.md`.

- **Financials-sector carve-out covers 23 companies**, not the spec's
  quoted 19 — sourced directly from the `sectors` table's `broad_sector =
  'Financials'` rows (banks, NBFCs, insurers, and financial holding
  companies); the exact roster is listed at the bottom of
  `ratio_edge_cases.log`.

- **The 8-pattern capital allocation classifier has an undefined 9th
  bucket.** The spec's 8 labels cover 7 of the 8 possible sign(CFO, CFI,
  CFF) combinations (splitting `(+,-,-)` into two by CFO/PAT quality).
  The remaining combination, `(-,+,-)`, isn't assigned a label by the
  spec; it's classified `"Unclassified"` (16 of 1,155 rows) rather than
  silently mis-bucketed into a defined pattern.

## Sprint 3 — Screener + Peer Engine: design decisions & notes for the next sprint owner

- **The screener universe is one row per company**, built by
  `src/screener/universe.py`: each company's latest standard-March-FYE
  `financial_ratios` row, left-joined with `profitandloss` (net_profit,
  sales — needed for two of the 15 filterable metrics that aren't in
  `financial_ratios`), `market_cap` (P/E, P/B, dividend yield, market
  cap — matched by calendar year, with a closest-available-year fallback
  for companies whose FYE falls outside `market_cap`'s 2019-2024
  coverage), `sectors`, `companies`, and `peer_groups`.

- **Two different "composite scores" now exist, on purpose.** Sprint 2's
  `composite_quality_score` (stored in `financial_ratios`, one value per
  company-*year*) was explicitly built as a provisional placeholder. Sprint
  3's `composite_score` / `composite_score_sector_relative` (computed at
  screening time by `src/screener/composite.py`, one value per company
  using the *latest* year) is the real Day-17 formula: P10/P90-winsorized,
  35% Profitability + 30% Cash Quality + 20% Growth + 15% Leverage. They
  intentionally disagree — the Sprint 2 version isn't wired into the
  screener, so drift between them is expected, not a bug.

- **Two preset thresholds were loosened from the literal spec numbers,
  documented in `config/screener_config.yaml`:**
  - *Value Pick* (`P/E<20, P/B<3.0`) matched only 2 companies against
    this dataset, whose Nifty 100 universe trades at a median P/E of
    ~46. Loosened to `P/E<40, P/B<7` -> 14 companies.
  - *Debt-Free Blue Chip* (`D/E==0` exactly) matched only 3 companies —
    real balance sheets almost always carry some minor lease/working-
    capital borrowing even when "debt-free" in spirit. Loosened to
    `D/E<=0.05` -> 16 companies.
  Both changes are commented inline in the YAML, not silently applied.

- **FCF CAGR isn't stored anywhere** (Sprint 2 only persists the
  per-year absolute `free_cash_flow_cr`) — `composite.py`'s
  `compute_fcf_cagr_5yr()` computes it on the fly per company using the
  same CAGR engine from Sprint 2, reusing `compute_cagr_for_window()`
  rather than duplicating CAGR logic.

- **`peer_percentiles` uses SQL-`PERCENT_RANK()` semantics**
  (`rank / (n-1)`, so the bottom of a group scores 0.0 and the top scores
  1.0), computed in pandas rather than SQL since SQLite's `PERCENT_RANK`
  window function support is version-dependent. A single-member peer
  group scores 1.0 by convention (there's no one to rank against).

- **Radar chart axes are normalised scores (0-100), not raw values** —
  plotting raw FCF (₹ Crore, in the hundreds/thousands) on the same axes
  as ROE (%) or D/E (a small ratio) would make every other axis
  invisible. The spec's own axis label ("FCF **score**", not "FCF Cr")
  confirms this; `radar.py` reuses `composite.py`'s winsorized scaling
  function for every axis except the pre-scored `composite_score` itself.

- **36 of 92 companies have no peer group** (only 56 companies across the
  11 peer groups in `peer_groups.xlsx`). These get a standalone two-bar
  chart (their composite score vs. the Nifty 100 average) instead of a
  radar, per the spec's Day 19 fallback — and are still counted, listed
  explicitly (not silently dropped) via `peer.no_peer_group_companies()`.
